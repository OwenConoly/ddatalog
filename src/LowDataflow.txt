- Every learned fact is broadcast to every node (including the sender).
- The exception is starred relations (which are mutable), which are internal to the node and are not broadcast anywhere.

rule 1.
// Pick any node, and put the following rule on it.
A(x) :- B(x)

rule 2.
// Let N be the number of nodes in the network.
// (for each i) On node i, put the following rules.
// |A|_i(s) means that the number of A-messages that node i will ever send is equal to s.

// this rule should be executed with a one-round delay, to ensure that all deducible A-facts have been sent already
// one-round delay means that if we learn a fact in round n, we cannot use it to fire this rule until round n + 2
// alternatively, could have some dummy auxiliary relation to enforce the delay.
|A|_i(s) :- |B|_i*(r, N), received(B)(r), sent(A)(s)

// for each j:
|B|_i*(x, 1) :- |B|_j(x)

rule 3.
// D* is a mutable relation.
// merge operation is addition, i.e. D*(x1, n1) + D*(x2, n2) = D*(x1 + x2, n1 + n2).
// in D*(x, n), the argument x is partial sum, n is number of elements summed so far.
// in |B|_i*(x, n), the argument x is partial sum (number of B-messages i should receive), and n is number of senders taken into account so far

// Pick any node, and put the following rules on it.
D*(x) :- A(x)

// to make sure D* is up-to-date with received A-facts, this rule gets a one-round delay.
D(x) :- D*(x), |A|*(x, N), received(A)(x)

// for each i:
|A|*(x, 1) :- |A|_i(x)

Extra features needed.
* one-round delay.  possible naive implementation: when adding new fact to trie, set some "old" bit to zero, then re-add fact to list of new facts for next round, when adding it in next round set old bit to 1.  alternative: keep track of whether we are on odd or even rounds, for each fact keep track of whether it was added on odd or even round.
* program stored on hardware may contain (in addition to normal relations), some list of data (sent|received) * relname * list (option nat), where the natural numbers should count upward from 1.  each element of the list gets a trie.  for instance, sent(R)(_, 1) means: before sending any R-message, look in the trie and increment the appropriate counter.  more comprehensible example: program above would have [(sent, A, nil), (received, B, nil)] on nodes executing rule 2, [(received, A, nil)] on nodes executing rule 3.
